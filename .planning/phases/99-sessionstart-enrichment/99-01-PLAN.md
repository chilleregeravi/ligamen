## Project hash algorithm (MUST match worker/db/database.js:75–82 projectHashDir)

```js
// Node reference (better-sqlite3 consumers):
function projectHashDir(projectRoot) {
  const hash = crypto
    .createHash("sha256")
    .update(projectRoot)
    .digest("hex")
    .slice(0, 12);
  return path.join(resolveDataDir(), "projects", hash);
}
```

Shell equivalent the enrichment block MUST use:

```bash
PROJECT_HASH="$(printf '%s' "$CWD" | shasum -a 256 | cut -c1-12)"
```

Notes:
- `shasum -a 256` is preinstalled on macOS AND Linux glibc images; if the executor finds a target without `shasum`, fall back to `sha256sum` (both emit `<hash>  -` so `cut -c1-12` works on either). Use `command -v shasum >/dev/null 2>&1 && HASHER=shasum || HASHER=sha256sum` at the top of the block.
- `printf '%s'` (no `-n`, no trailing newline) matches `crypto.createHash("sha256").update(projectRoot)` exactly — do NOT use `echo` (adds newline; produces a different hash).
- `$CWD` is already populated at line 29 of session-start.sh (with `$PWD` fallback).

## SQLite schema the queries depend on (from migration 001 + 005)

```sql
-- services: id, repo_id, name, root_path, language  (migration 001)
-- connections: id, source_service_id, target_service_id, protocol, method, path,
--              source_file, target_file  (migration 001)
-- scan_versions: id, repo_id, started_at, completed_at  (migration 005)
```

"Load-bearing files" operational definition (per research/ARCHITECTURE.md §273–314):
- DISTINCT non-null `connections.source_file` values (i.e. service-entry files the scan identified as the source of at least one outbound connection).

"Last scan date":
- `MAX(completed_at)` from `scan_versions` WHERE `completed_at IS NOT NULL`. Formatted as `YYYY-MM-DD` for the banner; the raw ISO timestamp is used for age calculations.

## hub.sh status --json output shape (from worker/cli/hub.js:111–148)

```json
{
  "plugin_version": "0.1.0",
  "data_dir": "/Users/.../.arcanon",
  "config_file": "/.../arcanon.config.json",
  "project_slug": "arcanon" | null,
  "hub_auto_upload": true | false,
  "credentials": "present" | "missing",
  "queue": { "pending": 0, "dead": 0, "oldestPending": null }
}
```

Hub banner status derivation (string rendered in banner):
- `credentials == "missing"` → `offline`
- `credentials == "present"` AND `hub_auto_upload == true` → `auto-sync on`
- `credentials == "present"` AND `hub_auto_upload == false` → `manual`
- Any error, non-zero exit, timeout, or JSON parse failure → `unknown`

NOTE: Phase 97 renames the key from `auto_upload` → `auto_sync` in hub.js (CLN-07) — Phase 99 runs AFTER Phase 97, so the two-read pattern in hub.js is already in place and `.hub_auto_upload` in the JSON output reflects whichever key is set. Do NOT re-plumb the rename here.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Claude Code runtime → session-start.sh | Claude Code invokes the hook with user-controlled `$CWD` in the stdin JSON. Treat `cwd` as attacker-influenced. |
| session-start.sh → local filesystem | Hook composes a DB path from `$CWD` + sha256(cwd). sha256 output is always safe-charset but `$CWD` is interpolated into string concatenation and passed to `sqlite3`. |
| session-start.sh → hub.sh subprocess | Hub.sh is a trusted bundled script; output is parsed by `jq`. The only external data comes from the local worker daemon. |
| sqlite3 → impact-map.db file | DB is read-only for this hook (SELECT COUNT only, no writes). A malicious or corrupt DB could crash sqlite3 — PRAGMA quick_check + 2>/dev/null + silent fallback absorbs this. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-01 | Tampering | Computed DB_PATH from `$CWD` | mitigate | `$CWD` is hashed via sha256 before being used in a filesystem path (`projects/<hash>/impact-map.db`), so path-traversal via `../` in cwd cannot escape `$DATA_DIR/projects/`. The hash is 12 hex chars — only `[0-9a-f]`. No shell-expansion risk. |
| T-99-02 | Denial of Service | sqlite3 queries hanging on a huge/corrupt impact-map.db | mitigate | Each query is wrapped in `2>/dev/null`; PRAGMA quick_check validates the file is SQLite before any real query runs; a malformed DB fails quick_check and is silent-fallback'd. Corrupt WAL or large map still completes a COUNT in ~milliseconds — counts are O(table rows). Additionally, `set -euo pipefail` means any unexpected failure unwinds the block via `|| true`. |
| T-99-03 | Information Disclosure | Enrichment banner leaking local paths into Claude context | accept | The banner emits only aggregate counts (`N services mapped`, `K load-bearing files`), a date, and a derived hub-status token. No `source_file` values, no project paths, no service names. The CWD is already visible to Claude (it sent it in the JSON input), so no new disclosure. |
| T-99-04 | Denial of Service | Hub.sh (Node cold-start) blocking session start on slow machine | mitigate | Hub.sh call output is captured with `2>/dev/null` and `|| HUB_JSON=""` so any hang, crash, or timeout falls through to `HUB_STATUS="unknown"`. The hook's 10s Claude Code–enforced timeout is the ultimate backstop. If hub.sh cold-start turns out to dominate the 200ms budget in production, a follow-up can shell out with `timeout 0.2 bash "$HUB_SH" …`; this is noted but not added now because macOS lacks `timeout` by default and the measured p99 in research was 50-100ms. |
| T-99-05 | Elevation of Privilege | jq parsing of hub.sh JSON with untrusted content | accept | Hub.sh output is from a local trusted Node CLI reading local config + local worker. jq is invoked with pure selector expressions (no `-f` script from external source). No interpolation risk. |
| T-99-06 | Spoofing | Fixture DB with arbitrary service counts injected by an attacker with write access to `~/.arcanon/projects/*/impact-map.db` | accept | Attacker with write access to the user's home directory already owns the session; enrichment correctness is not a trust boundary. |
| T-99-07 | Repudiation | Silent-fallback hides enrichment failures from the user — they can't tell if enrichment was skipped due to missing DB, corrupt DB, or bug | accept | Failure paths are intentionally silent per SSE-04 (session is never broken). For debugging, a future follow-up could add `ARCANON_SESSION_START_DEBUG=1` writing a trace to `$DATA_DIR/logs/` — out of scope for v0.1.1. |
