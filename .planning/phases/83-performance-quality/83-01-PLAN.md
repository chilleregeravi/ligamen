    - Test: After inserting 51 unique query patterns into the cache (capacity=50), the first pattern is evicted and the next call for it triggers a new db.prepare()
    - Test: Cache is keyed by the normalized SQL string, not by parameter values -- same SQL with different params reuses the statement
    - Test: search() results are unchanged (existing tests still pass)
  </behavior>
  <action>
    1. At the top of query-engine.js (after imports, before _searchDb), add a simple LRU cache class:
       - class StmtCache with constructor(capacity = 50)
       - Uses a Map (preserves insertion order) as the backing store
       - get(sql, db) method: if cache hit, delete+re-set to move to end (MRU), return stmt. If miss, call db.prepare(sql), evict oldest if at capacity, store and return.
       - clear() method to reset cache
       - size getter
       - Create module-level instance: const _stmtCache = new StmtCache(50)

    2. Replace the 3 inline db.prepare() calls in the FTS5 tier (lines ~88-97) with _stmtCache.get(sql, db). The SQL string literals stay identical -- just route through cache. Specifically:
       - services_fts MATCH query (line 89-96)
       Keep the .all(ftsQuery, perTable) call pattern on the returned statement.

    3. Replace the 1 inline db.prepare() call in the SQL tier (lines ~119-128) with _stmtCache.get(sql, db).

    4. Export _stmtCache for testing: export { _stmtCache }

    5. In query-engine-search.test.js, add a new describe block "search() -- prepared statement cache (REL-04)":
       - Import _stmtCache from query-engine.js
       - Test "reuses prepared statements for identical queries": clear cache, call search("payment", {skipChroma:true}) 100 times in a loop, then assert _stmtCache.size is small (1 or 2 statements, not 100). The FTS5 tier uses 1 SQL template, SQL tier uses 1 SQL template.
       - Test "LRU eviction at capacity": create a StmtCache(3) directly, insert 4 entries via get() with unique SQL strings, assert size === 3 and the first key is gone
       - Test "existing search results unchanged": already covered by existing tests -- just ensure they still pass

    IMPORTANT: The SQL string passed to _stmtCache.get() must be the template string (with ? placeholders), NOT the string with parameters substituted. This is how better-sqlite3 prepare() works -- the SQL is the cache key, parameters are passed to .all()/.get().
