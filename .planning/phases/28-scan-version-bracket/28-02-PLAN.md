the final connection that makes re-scan atomic: beginScan stamps the version ID
before agent invocation; endScan deletes stale rows after a successful persist.
Failed scans leave old data intact because endScan is never called on error paths.

// Returns integer scan version ID
beginScan(repoId): number

Current scanRepos loop structure to modify (worker/scan/manager.js):
```javascript
for (const repoPath of repoPaths) {
  const repo = queryEngine.upsertRepo({ path: repoPath, name: basename(repoPath), type: 'single' });
  const ctx = buildScanContext(repoPath, repo.id, queryEngine, options);

  if (ctx.mode === 'skip') {
    results.push({ repoPath, mode: 'skip', findings: null });
    continue;
  }

  // ... prompt interpolation ...
  const rawResponse = await agentRunner(interpolatedPrompt, repoPath);
  const result = parseAgentOutput(rawResponse);

  if (result.valid === false) {
    results.push({ repoPath, mode: ctx.mode, findings: null, error: result.error });
    continue;   // <-- endScan must NOT be called here
  }

  const currentHead = getCurrentHead(repoPath);
  queryEngine.setRepoState(repo.id, currentHead);    // <-- currently here
  results.push({ repoPath, mode: ctx.mode, findings: result.findings });
  // <-- endScan must be called here (success path only)
}
```

Note: scanRepos currently does NOT call persistFindings — it returns findings to the
caller. Check if persistFindings is called in the MCP server or HTTP handler that
invokes scanRepos. If scanRepos only returns findings (not persisting), then manager.js
cannot call endScan after persistFindings here. Re-read manager.js carefully before
implementing.

scanVersionId = qe.beginScan(repo.id)

Move the persistFindings call from the HTTP/MCP handler into scanRepos, and
remove it from the handler. The handler already receives the findings in the
ScanResult — it should not also call persistFindings.
