# Phase 113 — Verification Gate — Context

## Phase Goal

Final repo-wide grep returns clean. bats + node suites green. Manifest versions
bumped to 0.1.3. CHANGELOG `[0.1.3]` section pinned with all subsections
complete. Mirror of v0.1.2's Phase 105 (`105-VERIFICATION.md`) and v0.1.1's
Phase 100 verification — single plan + verification report.

This is the release pin for **v0.1.3 Trust & Foundations**. After this gate
passes, the milestone is ready for `/gsd-complete-milestone v0.1.3`.

**Requirements covered:** VER-01, VER-02, VER-03, VER-04, VER-05, VER-06, VER-07
(7 REQs).

## Decisions

### D-01 — Manifest count is 4, not 5

Previous milestones (v0.1.1, v0.1.2) bumped **5 manifest files** because
`plugins/arcanon/runtime-deps.json` was a manifest that carried its own version
string. v0.1.3 deletes `runtime-deps.json` entirely (INST-01, Phase 107), so
VER-06 expects exactly **4 files** at version 0.1.3:

1. `plugins/arcanon/.claude-plugin/plugin.json` — 1 occurrence
2. `plugins/arcanon/.claude-plugin/marketplace.json` — 2 occurrences (plugin
   entry + top-level)
3. `.claude-plugin/marketplace.json` (root) — 2 occurrences (plugin entry +
   top-level)
4. `plugins/arcanon/package.json` — 1 occurrence

**Total: 6 version strings across 4 files.**

`plugins/arcanon/runtime-deps.json` is **NOT** in the list — Phase 107 deletes
it. If this gate runs and the file still exists, Phase 107 didn't finish (the
VER-03 grep guard catches that case).

`plugins/arcanon/package-lock.json` is **NOT** a manifest in the VER-06 sense —
it is a generated lockfile. It carries 2 version strings that should match the
package.json bump, but those are regenerated by `npm install --package-lock-only`
(see D-02). The gate doesn't manually edit package-lock.json.

### D-02 — package-lock.json regeneration is part of the manifest bump

Lesson from v0.1.2 PR #19: bumping a dep version in `package.json` without
updating `package-lock.json` made CI's `npm ci` fail with a lockfile-out-of-sync
error. v0.1.3 doesn't bump any dep versions in package.json (better-sqlite3
stays at `^12.9.0` after v0.1.2's bump), but the **plugin's own `version`
field** moves from `0.1.2` → `0.1.3`, and that field also lives in
`package-lock.json` (twice — once as the root `version` and once as the
`packages.""` entry).

**Plan must include `npm install --package-lock-only` after the manifest bump**
so the lockfile's version fields line up with package.json before commit. Run
it from `plugins/arcanon/` so it touches the correct lockfile.

This is non-optional — skipping it reproduces v0.1.2's CI npm-ci breakage.

### D-03 — CHANGELOG date pins to phase execution date

Pin CHANGELOG `[0.1.3] - YYYY-MM-DD` to whatever `date -u +%Y-%m-%d` returns
**at the time Phase 113 executes**. Don't hardcode a date in the plan.

v0.1.2 used `2026-04-23` initially, then pushed to `2026-04-24` when hotfixes
folded in. For v0.1.3, the plan executor runs `date -u +%Y-%m-%d` as the first
step of the CHANGELOG-pinning task and substitutes the result.

If the gate ships in a single sitting (likely), the date is whatever that day
is in UTC. Plan instructs the executor to capture this once and reuse it.

### D-04 — `--help` regression check (VER-04) — zero strings expected

v0.1.4 will add a `--help` system across all commands (THE-1025 Item 2).
v0.1.3 explicitly **excludes** this — confirmed in REQUIREMENTS.md "Out of
Scope" table.

The VER-04 grep is a **regression guard**: if any plan in v0.1.3 accidentally
adds a `--help` string to a `commands/*.md` file, the gate catches it.

Since `commands/upload.md` is being deleted (DEP-01), and no other v0.1.3 work
introduces `--help`, the expected state is **zero `--help` strings in any
`commands/*.md` file**. Not "only allowed in deprecated stub" — literally zero.

### D-05 — `/arcanon:verify` (the command, Phase 112) is distinct from this verification gate (Phase 113)

Two different things share the word "verify":

- **`/arcanon:verify`** (Phase 112 work, TRUST-01) — a user-facing slash
  command that re-reads cited evidence in source files and returns
  ok/moved/missing/method_mismatch verdicts per connection. Read-only,
  user-invokable, idempotent.
- **Phase 113 Verification Gate** — an internal release-process phase that
  runs greps, test suites, and manifest checks before a milestone ships. Not
  user-facing.

The plan must NOT confuse them. The gate **does not run `/arcanon:verify`** as
part of its work — that's a Phase 112 deliverable verified by Phase 112's own
plan tests. The gate just runs greps, bats, node tests, and bumps manifests.

### D-06 — bats threshold env var: only needed on macOS dev runs, not in CI

The HOK-06 `IMPACT_HOOK_LATENCY_THRESHOLD=200` env override is needed **only
on macOS dev runs** because BSD fork overhead pushes p99 above the 50ms Linux
target. Documented v0.1.1 caveat.

CI's `.github/workflows/ci.yml` already sets `IMPACT_HOOK_LATENCY_THRESHOLD: "100"`
(committed in v0.1.1 milestone). Phase 113 **does not** modify the CI workflow
file. The plan instructs the executor to run bats locally with
`IMPACT_HOOK_LATENCY_THRESHOLD=200` and document the macOS caveat in the
verification report — same shape as `105-VERIFICATION.md`.

Expected bats count: **≥320 tests** (310 baseline from v0.1.2 + new tests added
in Phases 107-112: install-deps.bats from 107, verify.bats from 112, migration
tests from 109/110/111, regression bats from 108, plus a few). Plan documents
"≥310 acceptable, ≥320 expected" so a small under-count doesn't block the gate
unnecessarily.

  `105-VERIFICATION.md` line 63).
