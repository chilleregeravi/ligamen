  <action>
    **1. query-engine.js — Lower default maxDepth:**
    - Change `transitiveImpact` default from `maxDepth = 10` to `maxDepth = 7`
    - This is a one-line change at line 436

    **2. server.js — Add MAX_TRANSITIVE_DEPTH constant and update queryImpact:**
    - Add at module level: `const MAX_TRANSITIVE_DEPTH = 7;`
    - Add at module level: `const QUERY_TIMEOUT_MS = 30_000;`
    - In queryImpact, change the hardcoded `i.depth < 10` in the CTE to `i.depth < ${MAX_TRANSITIVE_DEPTH}`
    - After executing the CTE query, check if any row has depth === MAX_TRANSITIVE_DEPTH. If so, add a `truncated` field to the return:
      ```javascript
      const maxFound = Math.max(...rows.map(r => r.depth), 0);
      const truncated = maxFound >= MAX_TRANSITIVE_DEPTH;
      return {
        results,
        ...(truncated && { truncated: true, notice: `Results truncated at depth ${MAX_TRANSITIVE_DEPTH}` })
      };
      ```

    **3. server.js — Add 30s timeout wrapper:**
    - Wrap the transitive CTE execution in a timeout. Since better-sqlite3 is synchronous, use `db.pragma('busy_timeout', QUERY_TIMEOUT_MS)` is insufficient (that's for lock waits, not query duration).
    - Instead, use SQLite's `progress_handler` pattern if available, OR wrap in a Promise.race with setTimeout:
      ```javascript
      // For synchronous better-sqlite3, the simplest approach:
      // Set an interrupt timer and cancel it on completion
      const timer = setTimeout(() => { db.interrupt?.(); }, QUERY_TIMEOUT_MS);
      try {
        const rows = db.prepare(cte).all(serviceId, serviceId);
        clearTimeout(timer);
        // ... process rows
      } catch (e) {
        clearTimeout(timer);
        if (e.message.includes('interrupt')) {
          return { results: [], error: 'Query timeout: transitive impact query exceeded 30s', timeout: true };
        }
        throw e;
      }
      ```
      Note: Check if `db.interrupt` exists on better-sqlite3. If not available, use `db.function` progress handler or simply enforce via the depth limit (which is the primary guard). If interrupt is not available, document in code comment and rely on depth limit as the primary timeout prevention.

    **4. server.js — Update MCP tool schema (optional):**
    - Update the `transitive` field description from "up to depth 10" to "up to depth 7"

    - Test non-transitive query still works as before
