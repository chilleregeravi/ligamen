  <behavior>
    - When repoId is provided and a service with that name exists in that repo: return that service's ID (no warning)
    - When repoId is provided but the name does not exist in that repo, and exactly one service with that name exists globally: return that ID (legitimate cross-repo ref, no warning)
    - When repoId is provided and multiple services share that name across repos: return the first match and emit `console.warn` to stderr: `[ligamen] Ambiguous service name "${name}" matches ${count} repos — using id ${id}. Scope your connection to avoid collisions.`
    - When repoId is not provided (backward compat): existing behavior — SELECT first match globally, no warning
  </behavior>
  <action>
    1. Change _resolveServiceId signature from `_resolveServiceId(name)` to `_resolveServiceId(name, repoId = null)`.

    2. Replace the method body with the following logic (do NOT change the method's location in the file):
       a. If repoId is provided: query `SELECT id FROM services WHERE name = ? AND repo_id = ?` — if a row is found, return `row.id` immediately (same-repo exact match, no ambiguity possible).
       b. If no same-repo match (or no repoId given): query `SELECT id, repo_id FROM services WHERE name = ?` to get ALL matches.
       c. If zero rows: return null.
       d. If exactly one row: return `row.id` (unambiguous cross-repo reference, no warning needed).
       e. If multiple rows: emit `console.warn('[ligamen] Ambiguous service name "' + name + '" matches ' + rows.length + ' repos — using id ' + rows[0].id + '. Scope your connection to avoid collisions.')` then return `rows[0].id`.

    3. In persistFindings(), update both call sites to pass `repoId` as the second argument:
       ```js
       const sourceId =
         serviceIdMap.get(conn.source) || this._resolveServiceId(conn.source, repoId);
       const targetId =
         serviceIdMap.get(conn.target) || this._resolveServiceId(conn.target, repoId);
       ```
       `repoId` is already the first parameter of persistFindings — no additional lookup needed.

    Do NOT introduce any new prepared statements in the constructor for these queries; use inline `this._db.prepare(...).get(...)` / `.all(...)` calls inside the method (the method is called rarely — once per cross-repo connection edge — so no performance concern from non-cached statements).
  </action>
