Append the final branch to `extract_versions()` AFTER the NuGet branch. Parse ONLY `Gemfile.lock` (never `Gemfile` — ranges, not pins — Pitfall 3).

`if [[ -f "${repo_dir}/Gemfile.lock" ]]; then ... fi`:

Single awk state machine — track which of `GEM` / `GIT` / `PATH` section we're in, capture `specs:` indented entries. All three sections share the same `  gemname (version)` spec format:
```bash
awk '
  /^GEM$/    { section="GEM"; next }
  /^GIT$/    { section="GIT"; next }
  /^PATH$/   { section="PATH"; next }
  /^[A-Z]+$/ { section=""; next }          # any other top-level section (BUNDLED WITH, DEPENDENCIES, PLATFORMS) ends spec capture
  /^$/       { in_specs=0; next }
  section && /^  specs:/ { in_specs=1; next }
  # spec line: 4-space indent, name (version)
  in_specs && /^    [a-zA-Z0-9_-]+ \([0-9]/ {
    name=$1
    ver=$2
    gsub(/[()]/,"",ver)
    print name"="ver
  }
' "${repo_dir}/Gemfile.lock" 2>/dev/null | sort -u || true
```

Key awk details the executor must preserve:
- Top-level section lines are bare uppercase words (`GEM`, `GIT`, `PATH`, `DEPENDENCIES`, `BUNDLED WITH`, `PLATFORMS`). Match `/^GEM$/`, `/^GIT$/`, `/^PATH$/` explicitly; any other `/^[A-Z]+$/` line ends spec capture.
- `  specs:` line is 2-space indented, under the section header.
- Spec lines are 4-space indented with format `    name (version)` — transitive deps are 6-space indented, skipped by the `/^    [a-zA-Z0-9_-]+ \([0-9]/` anchor which requires exactly 4 spaces (awk default FS collapses whitespace but the regex anchor `^    ` enforces exact indent).
- Pipe through `sort -u` to dedup if the same gem appears in multiple sections (e.g. hot-reloaded from PATH but also pinned in GEM).
